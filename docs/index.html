What is this project?

Imagine you are a Mission Controller at a space agency. You have a team of specialized AI astronauts deployed to an unknown codebase. Your job isn't to physically write the code—it's to command them through your Mission Control dashboard (Streamlit) and watch as your specialists scout the files, implement logic, and run tests in isolated sandboxes, occasionally asking for your final authorization before doing anything dangerous.

This project builds a "Virtual Engineering Team" using CrewAI. Instead of asking ChatGPT one question at a time, we spawn different AI "Agents" with specific job titles, inject them into a dynamic workspace, and watch them collaborate in real-time through a Chat UI.

CodeHarness Autonomous Agents

Unlike standard LLMs that generate block text, CodeHarness leverages specialized autonomous agents equipped with distinct system prompts and sandbox permissions. They communicate directly with the central router to accomplish complex multi-step objectives.

🤖

The Manager

Clearance: Level 5 Orchestrator

The central brain of the swarm. The Manager ingests your natural language prompt, maps it to a repository objective, spins up dynamic workspaces, and delegates concurrent tasks to the specialized agents below.

Accessible APIs:
delegate_task()
review_PR()
ask_human()
🕵️

The Explorer

Clearance: Read-Only AST Mapper

The recon specialist. AI models hallucinate when blind. The Explorer rigorously maps the repository's Abstract Syntax Tree, greps for references, and reads file logic to locate precisely where modifications are needed before a single line of code is written.

Accessible APIs:
list_dir()
grep_search()
view_file()
👨‍💻

The Coder

Clearance: Write-Access Logic Synthesizer

The builder. Receives pinpointed instructions from the Manager and context from the Explorer to synthesize exact code replacements. It modifies files directly on disk, avoiding expensive full-file rewrites by using precise chunk replacements.

Accessible APIs:
multi_replace_content()
write_to_file()
🧪

The Tester

Clearance: Execution E2B Sandboxed

The verification engine. Spawns isolated secure environments to install dependencies, compile builds, and run unit tests. If a test fails, the Tester automatically loops the failure trace back to the Coder for continuous autonomous debugging.

Accessible APIs:
spawn_e2b_sandbox()
run_command()
manage_task()
🛡️

Security Interceptor

Clearance: Root (Human) Safeguard

The final defense. Hooks into the Swarm's tool execution layer and intercepts destructive operations (like terminal commands or overwrites), forcing the Swarm to halt and await your explicit cryptographic authorization.

Accessible APIs:
await_human_approval()
abort_execution()

The Safety Net (Human-in-the-Loop)

Allowing AI to write and execute code on your computer is like letting a robot swing a hammer in your house. It is powerful, but inherently dangerous.

SYSTEM EVENT: Gated Approval Hook Triggered
AWAITING AUTHORIZATION

To prevent chaos, this project uses a Gated Approval Hook mapped directly to the Streamlit UI layer. Before any AI agent is permitted to write to disk or run terminal commands, the framework forces a synchronous yield. The entire execution tree halts and queries you: "Approve this action?"

If you hit REJECT, the UI dispatches a HookAbortedError deep into the agent's logic flow, tearing down the malicious process, preserving the OS, and forcing the agent to rethink its strategy. It ensures absolute, impenetrable environment safety.

A Real-World Example: Issue #113021

To truly understand how smart the agents are, toggle between the Broken and Fixed code states below for the actual OpenClaw ACP subagent bug.

function reconnect(session) {
    session.childProcesses.forEach(p => {
        p.kill();
        logStatus(p.id, "stopped by user");
    });
}
The Problem:

When a parent session disconnects and reconnects, it aggressively kills all child processes and hardcodes the label "stopped by user". This orphans Background ACP subagents and incorrectly blames the user!

function reconnect(session) {
    session.childProcesses.forEach(p => {
        if (p.isACP) {
            p.orphan();
            logStatus(p.id, "orphaned");
        } else {
            p.kill();
            logStatus(p.id, "stopped by user");
        }
    });
}
The Solution:

The AI correctly parses the issue, identifies that ACP subagents need an "orphan recovery flow", adds a check to safely detach them, and updates the status label to "orphaned" as requested in the GitHub issue.

Dynamic Path Injection & Keys
import sys
from pathlib import Path
from dotenv import load_dotenv, set_key

root_dir = Path(__file__).resolve().parent.parent.parent.parent
sys.path.insert(0, str(root_dir / "src"))
🧠 Architect's Log When running Streamlit inside a nested directory layout, Python's absolute import resolution breaks. We dynamically compute the root path and inject src/ into sys.path at runtime so that imports resolve perfectly regardless of where the user executes the script from. We also use dotenv.set_key to ensure that API keys typed in the browser are securely persisted locally across reboots.

LLM Matrix (The Brain)
MODEL = os.getenv("MODEL", "gemini/gemini-2.5-pro")
if "groq" in MODEL or "gemini" in MODEL:
    llm = LLM(model=MODEL, api_key=...)
else:
    llm = LLM(model=MODEL)
🧠 Architect's Log We utilize the LiteLLM package to standardise requests across Anthropic, Google, and Groq. By using conditional instantiation, the system dynamically routes calls, bypassing strict json_schema incompatibilities that plague models like Groq's Llama 3.3. This ensures the agent is truly agnostic and immune to single-vendor deprecations.
The Safety Interceptor
from crewai.hooks import before_tool_call, HookAborted
GATED_TOOLS = {write_file.name, exec_tool.name}

@before_tool_call
def require_approval(context):
    if context.tool_name in GATED_TOOLS:
        if not ui_callback.ask_approval(context.tool_name):
            raise HookAborted(reason="User denied tool execution")
🧠 Architect's Log This is arguably the most critical code block in the entire project. The @before_tool_call hook intercepts the execution after the LLM decides to call it, but before the code actually runs. Execution is suspended, and the human is prompted via the UI. If rejected, we raise a deterministic HookAborted exception. This completely bypasses the LLM's non-determinism.

Tools Abstraction

src/codeharness/core/tools.py

Dual-Execution sandboxing logic to isolate terminal damage.

Dual-Execution Logic
if os.getenv("USE_E2B") == "true":
    from crewai_tools import E2BExecTool
    exec_tool = E2BExecTool()
else:
    from codeharness.core.tools import SubprocessTool
    exec_tool = SubprocessTool()
🧠 Architect's Log Enterprise security demands isolated VMs (E2B) for untrusted payloads. But independent developers need cost efficiency. By conditionally abstracting the exec_tool, the Tester agent doesn't know (or care) whether it's running tests in the cloud or in a local sandboxed subprocess. It just receives the standard stdout trace either way.
