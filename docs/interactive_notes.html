Rebuilding Claude Code's Harness

The gap between your agent and Claude Code isn't the model — it's the harness.

What is this project?

Imagine you run a restaurant, but instead of human chefs, you have a team of incredibly smart robot chefs. They need to cook a complex meal (fix a bug in software) without your constant supervision.

This project builds a "Virtual Engineering Team" using AI. Instead of asking ChatGPT one question at a time, we create different AI "Agents" with specific job titles, put them in a room, and tell them to solve a problem together.

Meet the Virtual Team

We use a framework called CrewAI to manage our team. Here are the specialized workers that make the autonomous workflow possible:

1. The Explorer (Scout)

Role: Look around the folders and read the files.

AI doesn't automatically know what files you have. This agent reads the blueprint before anyone starts building.

2. The Coder (Builder)

Role: Write and edit the code.

Once the Explorer finds the problem, the Coder physically "types" out the fix and saves the file.

3. The Tester (QA)

Role: Run the software and verify.

A coder can claim they fixed it, but the tester proves it by running automated tests in an isolated sandbox.

4. The Manager (Boss)

Role: Delegate and review.

The manager assigns the reading, the writing, and the testing, ensuring the overall objective is met.

The Safety Net (Human-in-the-Loop)

Allowing AI to write and execute code on your computer is like letting a robot swing a hammer in your house. It is powerful, but inherently dangerous.

To prevent chaos, this project uses a Gated Approval Hook. Before the AI is allowed to save a file or run a command, a prompt pauses execution to ask a human: "Approve this action? [Yes/No]". If denied, the execution is abruptly aborted at the framework level via a HookAborted exception, ensuring absolute safety.

Top-Level Interview Questions

If you were discussing this architecture in a Staff/Senior Engineering interview at a top tech company, these are the questions you would face, paired with top-tier technical responses.

System Design & Architecture

1. Why use a multi-agent system instead of a single LLM with tools?

Response:

"A single LLM agent suffers from context degradation and objective drift when handling complex, multi-step tasks. By utilizing a multi-agent architecture via CrewAI, we implement the Separation of Concerns pattern at the cognitive level."

"We decouple the 'reading' context from the 'writing' context. If one agent does everything, its prompt context window becomes saturated with file reads, causing it to hallucinate or forget the core objective. By explicitly defining an Explorer, Coder, and Tester, each agent has a strictly bounded context and a specialized system prompt. A Manager agent acts as a semantic router, delegating tasks and acting as a verification loop. This drastically reduces hallucination rates and increases determinism."

Security & Sandboxing

2. How did you secure the environment against malicious execution?

Response:

"Executing LLM-generated code is inherently a remote code execution (RCE) vulnerability. To mitigate this, I implemented a defense-in-depth strategy with two primary layers."

"First, I utilized E2B (Ephemeral 2 Sandbox) for isolated execution. The Tester agent does not run commands on the host machine; it runs them inside an ephemeral, containerized micro-VM."

"Second, I implemented a Control Plane Interceptor using a strict Human-in-the-Loop (HITL) gate. When high-risk tools are invoked, the hook intercepts the payload and blocks the thread. Crucially, if approval is denied, the system raises a `HookAborted` exception, forcibly aborting the execution at the framework level rather than just asking the LLM nicely."

State Management & Resilience

3. How do you handle scenarios where the Tester reports a failure?

Response:

"The workflow isn't a linear DAG; it's a feedback loop. When the Tester agent runs the test suite via the sandbox, the `stdout` and `stderr` (e.g., the Pytest traceback) are returned."

"The Manager evaluates this result. Because the objective explicitly states 'iterate until green', a test failure triggers a state transition back to the Coder agent. Furthermore, by utilizing short-term memory embeddings, the Coder remembers its previous failed attempts, preventing infinite loops of identical incorrect changes."

Scalability

4. How would you scale this to a monorepo?

Response:

"Currently, the Explorer uses raw directory reading, which has an $O(N)$ complexity regarding token usage and would quickly exceed the LLM context limit. To scale, we must move from read-based discovery to search-based discovery."

"I would implement three systems: 1. Semantic Search (RAG): Embed the codebase using a vector database. 2. AST Indexing: Provide tools like LSIF so the agent can accurately find function definitions. 3. Chunked Editing: Instead of overwriting entire files, I would provide a `patch` tool, allowing the Coder to mutate only specific lines, vastly reducing generation latency."

Behavioral & Debugging

5. Tell me about a time this project failed and how you debugged it.

Response:

"During development, I noticed the Human-in-the-Loop permission layer was failing to block unauthorized actions. The agent proceeded even when I typed 'no'. It looked like an LLM hallucination—as if the model was ignoring the 'denied' response."

"Instead of prompting the model harder, I dug into the framework's internal execution state. I discovered a bug in the `@before_tool_call` hook: the code simply used `return False`. However, returning a boolean does not halt the thread; the framework expects a hard exception to short-circuit the execution stack. I fixed this by raising the framework's native `HookAborted` exception. This reinforced my core philosophy: Treat the LLM as an unreliable runtime environment, and build safety gates in deterministic code."

🏗️ The 6-Layer Architecture

CodeHarness replaces a fragile while-loop with a robust, six-layer architecture designed for production-level AI engineering.

  • Core Loop

    The fundamental Task → Tool → Result execution cycle, orchestrated cleanly by the CrewAI engine.

  • Tools Layer

    Provides agents with native, deterministic access to read/write files and execute sandboxed shell commands.

  • Planning Engine

    A strategic roadmap generated before coding begins, preventing "context rot" during execution.

  • Subagent Delegation

    A semantic routing Manager delegates discrete tasks to specialists (Explorer, Coder, Tester) to drastically save context space.

  • Sandboxing (E2B)

    Untrusted LLM code and test suites execute inside an isolated, containerized micro-VM (E2B), protecting the host machine.

  • Memory & Checkpointing

    Long-term embeddings persist facts across sessions, allowing the agent to survive interruptions and remember past failures.

🛡️ Safety First

If you are building agents, remember a core law of AI Engineering: Prompts are not safeguards. Sandboxes and Interceptors are.

CodeHarness forces all untrusted code execution through E2B, ensuring your host machine remains pristine. Furthermore, it employs a Human-in-the-Loop hook (@before_tool_call) that requires explicit human approval via stdin before any high-risk file writes or system executions. If denied, it uses a deterministic HookAborted exception to halt the agent stack immediately.

Initialization & Discovery

The system connects to OpenRouter and E2B via your `.env` keys. The Manager Agent parses the problem and dispatches the Explorer to read `account.py` using its native file-read tools.

2

Contextual Coding

With perfect context loaded into its memory, the Coder agent computes the fix. It determines exactly which lines need mutation to resolve the bug, without rewriting the entire file or hallucinating new variables.

3

The Interceptor (You)

The Coder attempts to save the mutation to your local disk. Execution immediately freezes. The framework prompts you: Approve write to account.py? [y/N]. The agent cannot proceed until you grant manual authorization.

4

Sandboxed Verification

Once authorized, the code is updated. The Tester takes over, instantly spinning up an ephemeral E2B cloud sandbox. It injects the codebase and securely runs `pytest`. If it fails, the trace is sent back to the Manager for a new iteration. If it passes, the task is complete.

A Real-World Example (For Beginners)

To truly understand how smart the agents are, let's look at the actual code bug they fix in the demo, explained simply.

❌ The Broken Code (Before)
def withdraw(self, amount):
    self.balance -= amount
Beginner Translation: The code blindly subtracts money from your account. It's like a broken ATM that lets you withdraw $1,000 even if you only have $50 in your account!
✅ The Agent's Fix (After)
def withdraw(self, amount):
    if amount > self.balance:
        raise ValueError("No funds")
    self.balance -= amount
Beginner Translation: The AI automatically spots the problem and adds a safety check (the green text). Now, the ATM checks your balance first, and stops the transaction if you don't have enough money.

Codebase Architecture & Line-by-Line Breakdown

A comprehensive, granular deep dive into the Python backend. This guide explains the psychological and architectural reasoning behind every significant line of code in the project, preparing you to defend it in an intense technical interview.

Core Engine

1. code_harness.py

The primary execution engine. It configures the language model, tools, the Agent swarm, and enforces the crucial human-in-the-loop safety interceptor.

1.1 Imports & Environment setup

import os
from dotenv import load_dotenv
load_dotenv()
from crewai import LLM, Agent, Crew, Process, Task
from crewai_tools import DirectoryReadTool, FileReadTool, FileWriterTool

Thought Process: Environment variables (API keys for Claude, OpenAI, E2B) must never be hardcoded. python-dotenv safely loads them from a local .env file that is .gitignored. We import CrewAI because it provides native support for hierarchical orchestration (Manager delegating to workers) and advanced tool handling.

1.2 Model Selection (The Brain)

MODEL = os.getenv("MODEL", "openrouter/anthropic/claude-sonnet-4-6")
llm = LLM(model=MODEL)

Thought Process: Instead of locking the application into a single vendor (like OpenAI), we route the LLM request through OpenRouter. This allows us to instantly swap models (e.g., from GPT-4o to Claude 3.5 Sonnet) just by changing the .env file. Claude 3.5 Sonnet is specifically chosen here because it currently leads industry benchmarks in zero-shot coding tasks.

1.3 Generic File Tools (The Hands)

read_file = FileReadTool()
write_file = FileWriterTool()
list_dir = DirectoryReadTool()
filesystem_tools = [read_file, write_file, list_dir]

Thought Process: These are native CrewAI tools. By explicitly isolating read tools from write tools, we can grant agents the Principle of Least Privilege (e.g., the Explorer only gets read tools, so it is physically impossible for it to accidentally delete code while scouting).

1.4 E2B Sandbox Integration (The Cloud Container)

sandbox_tools = []
exec_tool = None
if os.getenv("E2B_API_KEY"):
    from crewai_tools import E2BExecTool, E2BPythonTool
    exec_tool = E2BExecTool()
    sandbox_tools = [exec_tool, E2BPythonTool()]

Thought Process: This is the most impressive architectural feature. Allowing an AI to execute arbitrary code on a host machine is a massive security risk. We integrate E2B, which dynamically spins up ephemeral, isolated micro-VMs in the cloud. If the AI hallucinates a destructive rm -rf / command, it only destroys a disposable cloud container, protecting the host machine perfectly. The if statement elegantly disables this if the API key is missing.

1.5 The Custom Test Wrapper

@tool("run_tests")
def run_tests(path: str = "tests/") -> str:
    """Run the pytest suite at the given path inside the sandbox and return the result."""
    return exec_tool.run(command=f"pytest {path} -q")
custom_tools = [run_tests]

Thought Process: LLMs struggle with open-ended "execute terminal commands" tools because the state space is infinite. By wrapping the generic exec_tool into a highly specific run_tests tool with strict parameters, we significantly constrain the AI's action space. It can only execute pytest, and it automatically routes execution through the E2B sandbox.

1.6 Defining the Swarm (Agents)

explorer = Agent(
    role="Codebase Explorer",
    goal="Map the repository and surface the files relevant to the task.",
    backstory="You never guess at file contents -- you read them.",
    tools=[read_file, list_dir],
    llm=llm
)

coder = Agent(
    role="Software Engineer",
    goal="Implement the requested change by editing files on disk.",
    tools=filesystem_tools + sandbox_tools,
    reasoning=True,
    llm=llm
)

Thought Process: In agentic systems, the backstory serves as the crucial System Prompt tuning reliability.

- Explorer: Gets only read tools. Its job is to build a massive context window of the current code state without mutating anything.
- Coder: Gets write tools and sandbox tools. Crucially, reasoning=True is enabled. This acts as a Chain-of-Thought forcing function. The Coder must explicitly draft a reasoning plan ("I will edit line 23 because...") before it is allowed to execute the write_file tool, drastically reducing careless hallucinations.

1.7 The Orchestrator (Manager)

manager = Agent(
    role="Engineering Lead",
    goal="Break the request into steps and delegate each to the right specialist.",
    llm=llm,
    allow_delegation=True
)

Thought Process: The manager is the router. allow_delegation=True enables the hierarchical swarm. The manager never writes code; it reads the Task objective, assigns the Explorer to scout, reviews the Explorer's output, commands the Coder to execute, and commands the Tester to verify. If the Tester returns an error trace, the Manager routes the failure back to the Coder for another iteration.

1.8 The Safety Interceptor (Human-in-the-Loop)

from crewai.hooks import before_tool_call, HookAborted
GATED_TOOLS = {write_file.name, "run_tests", *(t.name for t in sandbox_tools)}

@before_tool_call
def require_approval(context):
    if context.tool_name in GATED_TOOLS:
        response = input(f"Approve {context.tool_name} with input {context.tool_input}? [yes/no] ")
        if response.strip().lower() != "yes":
            raise HookAborted(reason="User denied tool execution", source="require_approval")

Thought Process: This is arguably the most critical code block in the entire project. LLMs are non-deterministic; they can and will go rogue. System prompts ("Do not delete files") are mere suggestions to a neural net. Prompts are not safeguards.

We define a set of GATED_TOOLS (anything that mutates disk or executes code). The @before_tool_call hook intercepts the execution after the LLM decides to call it, but before the code actually runs. Execution is suspended, and the human is prompted via stdin. If the human types "no", we raise a deterministic HookAborted python exception. This completely bypasses the LLM's non-determinism, forcing a hard halt. It represents true enterprise-grade AI safety.

1.9 Crew Instantiation & Memory

crew = Crew(
    agents=[explorer, coder, tester],
    tasks=[task],
    manager_agent=manager,
    process=Process.hierarchical,
    planning=True,
    planning_llm=LLM(model="openrouter/openai/gpt-4o-mini"),
    memory=True,
    checkpoint=True
)

Thought Process:
- Process.hierarchical turns the array of agents into a managed swarm rather than a sequential list.
- planning=True injects an AgentPlanner that writes a step-by-step logical plan before the execution loop begins. We use gpt-4o-mini for this specifically because planning requires deep reasoning but not heavy coding, making it highly cost-effective.
- memory=True enables cross-session embeddings (RAG), meaning the swarm remembers facts from previous runs.
- checkpoint=True allows the state to serialize to disk, so if a run crashes, it resumes from the checkpoint instead of starting over.

Target Application

2. Workspace/account.py

The target codebase representing the buggy banking application. The AI is challenged to find and resolve semantic logic errors within this object-oriented structure.

2.1 Custom Exceptions

class InsufficientFunds(Exception):
    pass

Thought Process: When designing enterprise backend logic, relying on generic Exception or ValueError for everything is an anti-pattern. By creating custom exceptions like InsufficientFunds, downstream services (like a web API) can catch exactly what went wrong and return precise HTTP 400 status codes. The AI must understand this to fix the code properly.

2.2 Object-Oriented State

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance
        self.history = []

Thought Process: This demonstrates clean encapsulation. The state (balance, history ledger) is tightly bound to the object instance. The AI must realize that it cannot mutate self.balance without also appending a record to self.history, testing its ability to handle multi-step state mutations.

2.3 The Business Logic & Bugs

def withdraw(self, amount):
    if amount <= 0:
        raise ValueError("amount must be positive")
    if amount > self.balance:
        raise InsufficientFunds("insufficient funds")
    self.balance -= amount
    self.history.append(("withdraw", amount))

Thought Process: The original bug in the demo omitted the boundary check if amount > self.balance, meaning accounts could be overdrawn into negatives. The AI Coder is challenged to contextually insert this specific boundary check and raise the custom exception without breaking the ledger appending mechanism below it.

Verification Layer

3. Workspace/tests/test_account.py

The deterministic pytest suite. This acts as the absolute source of truth that prevents the AI from hallucinating a successful fix.

Thought Process: Explain Test-Driven Development (TDD) in the context of Autonomous AI. Large Language Models are known for "sycophancy" (agreeing with the user or claiming success when none exists). If the agent simply says "I fixed the bug," you cannot trust it.

By providing a test suite, you strip away the LLM's ability to lie. The orchestration loop commands the Tester agent to run pytest inside the E2B sandbox. The output trace (which is completely deterministic) is parsed by the Manager. If there are red failing tests, the Manager automatically routes the execution back to the Coder for another attempt. It will iterate relentlessly until the objective output is entirely green.
